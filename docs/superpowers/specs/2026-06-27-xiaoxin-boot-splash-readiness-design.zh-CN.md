# 小新启动动画退场与透明背景设计

## 背景

小新 1.46 固件当前有开机 GIF 启动层。用户希望启动动画不要等到 Wi-Fi 或激活流程完成才消失，而是在正常 UI 已经可以显示时退场。同时，启动 GIF 只显示 logo 内容，logo 之外保持屏幕纯黑。

## 目标

1. 启动动画的退场条件改为“主 UI 已经建立并可以显示正常界面”。
2. 启动动画不依赖 Wi-Fi 连接成功、OTA 检查完成或激活完成。
3. 启动画面除 logo/GIF 可见内容外为纯黑背景。
4. 避免启动动画提前消失导致短暂空白、白底或未初始化 UI 闪烁。

## 非目标

1. 不改变 Wi-Fi、OTA、激活流程本身。
2. 不改变普通桌宠 GIF 的渲染逻辑。
3. 不引入桌面模拟器。本次只修正板端启动视觉行为。

## 现状

`PaopaoPetDisplay::SetupUI()` 会建立主屏幕、宠物帧缓冲、卡片层、低功耗时钟层、通知浮层，然后调用 `PlayGifState(current_state_)` 和 `ShowBootSplashLocked()`。启动动画层在最上方显示。

`Application::HandleActivationDoneEvent()`、激活码显示和 Wi-Fi 配网模式中会调用 `display->CompleteBootSplash()`。这使得启动动画在部分路径上和网络/激活流程耦合过深。

`main/assets/images/boot.gif` 当前没有透明通道，角落像素接近白色。因此即便启动层背景是黑色，GIF 自身仍会带出非黑背景。

## 设计

### 启动动画退场

在小新 1.46 的显示实现中新增“正常 UI 首帧就绪”的退场路径。推荐位置是 `PaopaoPetDisplay::SetupUI()` 完成以下步骤之后：

1. 宠物帧缓冲初始化完成。
2. `pet_image_` 已经创建并绑定到 `pet_frame_dsc_`。
3. 卡片、时钟、通知浮层完成初始化。
4. `PlayGifState(current_state_)` 已经准备好默认桌宠状态。
5. `ShowBootSplashLocked()` 已经显示启动层。

之后安排一次短延迟或 LVGL 定时器回调调用 `CompleteBootSplash()`，让底下的正常 UI 有机会完成至少一次刷新。这样启动动画不会等 Wi-Fi，但也不会在正常 UI 还没画出来时直接消失。

保留现有 `CompleteBootSplash()` 调用点作为幂等兜底。重复调用应安全返回，不产生副作用。

### GIF 透明背景

优先修正资产：将 `main/assets/images/boot.gif` 中接近白色的背景抠成透明，保留 logo 和动画主体。启动层继续使用纯黑 `boot_splash_layer_`，最终视觉效果为黑底上显示透明 GIF 内容。

如果资产处理后仍有边缘残留，再考虑只对启动 GIF 增加专用解码参数，把接近白色的像素视为透明或黑底合成。但这不是首选，因为普通 GIF 渲染不应被启动页需求污染。

## 错误处理

1. 如果启动 GIF 加载失败，保留当前行为：隐藏启动层并显示正常 UI。
2. 如果退场定时器创建失败，直接隐藏启动层，避免长期遮挡 UI。
3. 如果正常 UI 初始化失败，不新增特殊恢复流程，沿用现有日志和启动诊断。

## 测试与验证

1. 用 Python 检查 `boot.gif` 至少部分像素 alpha 为 0，角落像素透明。
2. 运行现有启动路径守护测试，确保启动诊断和小新板卡路径未回退。
3. 构建固件，确认 GIF 资产被重新嵌入。
4. 烧录后观察：开机短暂显示 logo 动画，正常桌宠 UI 出现后动画消失；未连接 Wi-Fi 时也不长期遮挡 UI。

## 备注

当前工作区已有其它未提交固件改动。本设计只约束后续启动动画相关修改，不要求清理或重写已有改动。
