# 小芯通知即时浮层规划

> 日期：2026-06-24
> 范围：Waveshare ESP32-S3 Touch LCD 1.46 小芯/泡泡宠物固件
> 目标：为所有进入通知中心的通知增加统一的视觉即时浮层，让用户在通知发生当下就能感知，而不是只能事后进入通知页查看。

## 背景

当前通知中心已经具备真实事件仓库、通知卡片、优先级排序、单条清理、全部清理和部分真实事件接入。低电量、WiFi 断开/配网中、语音识别失败、课程提醒等事件已经可以进入通知中心。

现有问题是：通知只躺在通知页面里。用户如果没有主动滑到通知页，就很难意识到刚才发生了新的通知。手机、手表等设备通常会在通知发生时先弹出即时提示，再允许用户之后进入通知中心查看历史。小芯也需要这一层即时反馈。

## 设计目标

- 所有进入通知中心的通知，在发生当下都显示统一的视觉即时浮层。
- 所有通知类型使用同一种浮层强度、尺寸、动画和停留时长，体验接近手机等电子设备的通知横幅。
- 通知类型只影响图标、标题、正文、通知中心留存时间和状态恢复策略，不影响即时浮层的强度。
- 本轮只规划视觉弹窗，不加入提示音、震动或其他触觉反馈。
- 通知中心卡片未来需要显示通知时间，但不作为即时浮层第一轮实现的必要前置。

## 非目标

- 不做提示音、震动、语音播报。
- 不为不同通知类型设计不同强度的弹窗。
- 不把普通聊天回复重新纳入通知中心。
- 不在本轮引入复杂通知操作，例如展开、回复、按钮确认。
- 不改变 Home、Notifications、Overview 三页的核心分页结构。

## 用户体验

通知事件发生时，当前页面不切走。屏幕上出现一个统一的通知浮层，展示图标、标题和一行正文。浮层停留固定时长后自动收起，同时通知已经写入通知中心。

示例：

```text
语音识别失败
刚才没听清
```

```text
WiFi 已断开
请检查网络
```

```text
OTA 更新
发现新版本
```

统一行为：

- 浮层展示时长固定，建议第一版为 3 秒。
- 浮层不抢占页面，不自动跳转到通知中心。
- 多条通知连续发生时按队列顺序展示，后一条等待前一条收起后再显示。
- 同一类型状态通知重复 upsert 时，如果标题或正文发生变化，可以再次触发浮层；内容完全相同的短时间重复事件可合并，避免连续刷屏。
- 用户手动进入通知页后，通知中心仍按现有卡片逻辑展示留存通知。

## 视觉规则

即时浮层采用统一视觉组件，建议优先使用当前系统浮层和底部字幕已有的风格，减少新样式分叉。

第一版规则：

- 位置：屏幕上方固定横幅，接近手机通知横幅，也避免和项目已有底部字幕区域冲突。
- 内容：一枚小图标、标题、一行正文。
- 尺寸：适配 1.46 寸圆屏，最多显示两行文字。
- 动画：短滑入或淡入，停留后淡出。
- 文本溢出：标题和正文都需要截断或缩短，不允许撑破圆屏。
- 层级：高于 Home、Notifications、Overview、设置页和低功耗时钟层。

低功耗时钟场景：

- 如果设备处于当前项目的低功耗时钟/AOD 页面，通知仍显示同一种视觉浮层。
- 浮层收起后返回低功耗时钟，不切回完整 Home 页面。
- 真实深睡或屏幕完全关闭状态不在第一版范围内；当前项目主要处理的是低功耗时钟覆盖层。

## 通知链路

统一链路如下：

```text
外部事件发生
-> 构建 xiaoxin_notification_event_t
-> 写入通知中心仓库
-> 派发即时浮层请求
-> UI 队列展示浮层
-> 根据 ttl_ms 或状态恢复策略清理通知中心卡片
```

即时浮层和通知中心共用同一个通知事件事实源，避免出现“浮层说一套，通知页显示另一套”的状态分叉。

## 模块边界

建议新增或抽出一个轻量通知浮层模块，避免继续膨胀板级显示文件。

建议职责：

- 通知中心仓库：继续负责通知事件 upsert、remove、排序、清理和 TTL 留存。
- 即时浮层控制器：负责接收通知事件、排队、去重、控制显示时长和动画状态。
- LVGL 显示层：负责创建浮层对象、更新标题/正文/图标、执行显示和隐藏动画。

显示层不应该直接决定通知是否留存，也不应该绕过通知中心单独生成另一套通知内容。

## TTL 与留存策略

即时浮层是否展示，与通知是否长期留存在通知中心是两件事。所有通知都展示浮层，但通知中心留存由事件类型决定。

第一版建议：

| 通知类型 | 即时浮层 | 通知中心留存 |
| --- | --- | --- |
| 语音识别失败 | 显示 | 使用短 TTL，例如 8 秒 |
| WiFi 断开/配网中 | 显示 | 状态恢复后移除 |
| 低电量 | 显示 | 电量恢复后移除 |
| OTA 更新/失败 | 显示 | 完成、恢复或用户清理后移除 |
| 课程/提醒 | 显示 | 按提醒策略留存或用户清理 |

`ttl_ms` 的补完仍然重要。它负责通知中心里的短期通知自动消失；即时浮层只负责发生当下提醒。

## 通知时间

通知中心卡片后续需要显示通知发生时间，例如：

```text
刚刚
3 分钟前
14:05
```

这需要为通知事件或卡片仓库补充创建时间字段，并在通知卡片渲染时显示。该能力和即时浮层相关，但可以作为后续独立小步推进：

1. 通知事件进入仓库时记录 `created_at` 或 `created_at_ms`。
2. 通知卡片显示相对时间或当天时间。
3. `upsert` 同类型通知时根据业务决定保留首次时间还是更新时间。

第一版即时浮层可以先不显示时间，因为浮层本身代表“刚刚发生”。

## OTA 通知接入

OTA 事件已经有预留类型，后续应接入真实流程：

- 检测到可用更新：写入 OTA 通知并显示即时浮层。
- 下载或升级中：upsert 同一 OTA 通知，并显示即时浮层。
- 下载失败或升级失败：upsert 失败通知，并显示即时浮层。
- 升级完成或无需用户处理：移除 OTA 通知。

OTA 通知不需要特殊强度，仍使用统一浮层。

## 错误处理

- 如果浮层对象创建失败，通知仍必须进入通知中心。
- 如果连续通知过多，浮层队列应有上限；超过上限时保留最新通知，避免内存无限增长。
- 如果通知正文为空，浮层只显示标题。
- 如果标题为空，使用通知类型默认标题。
- 如果通知中心 upsert 被忽略，例如普通聊天回复兼容类型，不能触发即时浮层。

## 测试策略

建议覆盖以下路径：

- 所有可进入通知中心的通知事件都会触发即时浮层队列。
- 普通聊天回复不会触发即时浮层。
- 语音识别失败显示浮层，并按 `ttl_ms` 从通知中心自动过期。
- WiFi 断开显示浮层，连接恢复后通知中心移除。
- OTA 状态变化使用同一通知类型 upsert，并每次有效状态变化触发浮层。
- 连续两条通知按队列顺序展示。
- 低功耗时钟页面上收到通知时，浮层显示后回到低功耗时钟。
- 通知卡片未来时间字段有独立测试，不阻塞第一版浮层。

## 实施顺序建议

1. 给通知事件仓库补 TTL 到期清理能力，先让短期通知生命周期正确。
2. 定义即时浮层队列和统一展示规则，接到通知事件 upsert 成功后入队。
3. 在 Waveshare 1.46 板级 LVGL 层创建统一浮层对象，接入显示/隐藏动画。
4. 接入现有通知事件：低电量、WiFi、语音识别失败、课程提醒。
5. 把 OTA 检测、下载、失败等真实流程接入 `XIAOXIN_NOTIFICATION_EVENT_OTA_UPDATE`。
6. 后续独立补通知卡片时间显示。

## 验收标准

- 任意真实通知发生时，用户无需进入通知页也能看到统一视觉浮层。
- 所有通知类型的浮层强度一致，只有内容不同。
- 浮层自动收起后，当前页面保持原状态。
- 可留存通知能在通知中心看到；短期通知能按 TTL 自动消失。
- OTA 真实状态能进入通知中心并触发统一浮层。
- 不引入提示音或震动。
